# Claude-Mem Local v13.11.0-local.3

## What this release is

`v13.11.0-local.3` makes Claude-Mem follow the model route of the active Claude Code session through the companion [`satan9394/cc-switch_mem`](https://github.com/satan9394/cc-switch_mem) fork. It preserves ordinary Claude Code model names and applies the `MEM` label only to Claude-Mem traffic.

## Key behavior

- Claude-Mem follows the resolved model for the same Claude session, including the first request after `/model`.
- If Claude Code resolves to `deepseek-v4-flash`, Claude-Mem uses `deepseek-v4-flash` too.
- Claude Code's original request model and existing CC Switch mapping remain unchanged.
- Claude-Mem requests continue to appear as `MEM` in CC Switch usage details.
- Missing session routing state fails locally with HTTP 409; there is no Opus, Sonnet, fixed-model, account, or provider fallback.

## Local security boundary

- Claude-Mem sends CC Switch only a local session identifier in addition to the existing `claude-mem` source marker.
- CC Switch keeps session-to-model state only in process memory: at most 1024 entries, each expiring after two hours.
- The follow-session header is removed before the request is forwarded upstream.
- Model prompts may still leave the machine through the provider selected in CC Switch. Local memory databases, indexes, and provider-audit metadata remain local.
- Cloud Sync, cmem.ai upload, PostHog telemetry, and silent provider fallback remain disabled.

## Upstream policy

The reviewed upstream base remains `thedotmack/claude-mem` `v13.11.0`. Upstream changes are advisory and are not automatically trusted or merged; local safety constraints take priority.
