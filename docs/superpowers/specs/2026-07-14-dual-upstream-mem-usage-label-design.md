# Dual-Upstream Maintenance and MEM Usage Label Design

**Date:** 2026-07-14
**Status:** Approved in conversation
**Applies to:** `satan9394/claude-mem_local` and a user-controlled CC Switch fork

## Outcome

Claude-Mem model requests remain semantically identical to ordinary Claude Code requests: the original `model` and `request_model` values, provider selection, model mapping, routing, token accounting, and pricing behavior do not change.

Only requests generated by Claude-Mem carry a local attribution marker. CC Switch consumes that marker before upstream forwarding and records the request's usage source as `MEM`. The existing CC Switch usage table therefore distinguishes Claude-Mem consumption without changing the model name:

| Traffic | Model shown by CC Switch | Source shown by CC Switch |
|---|---|---|
| Ordinary Claude Code | Original model value | `proxy` |
| Claude-Mem | Original model value | `MEM` |

Claude-Mem Local also tracks stable releases from both `thedotmack/claude-mem` and `farion1231/cc-switch`. Detection is automatic; synchronization, merging, installation, and publication remain security-gated human decisions.

## Options considered

Three attribution mechanisms were considered:

1. **Local request-source contract (selected).** Claude-Mem sends a narrowly defined header; CC Switch consumes it locally and stores `MEM` in the existing `data_source` field. This preserves all model semantics and uses the existing Source column.
2. **Rewrite CC Switch's SQLite rows after each request.** This avoids changing CC Switch but couples Claude-Mem to another application's private database, risks lock contention and partial updates, and cannot reliably label failure rows. It is rejected.
3. **Suffix the model or duplicate the provider.** A model suffix can alter routing, compatibility, and price lookup. A provider rename affects ordinary Claude Code traffic sharing that provider. Both are rejected.

The selected design is a small two-repository compatibility contract. It requires no new runtime dependency, database, queue, or background service.

## Attribution protocol

Claude-Mem's CC Switch provider adds exactly this request header:

```http
x-cc-switch-usage-source: claude-mem
```

The header is added only in `cc-switch-auto` mode and only after the destination has passed the existing loopback policy. Direct official providers, legacy providers, Viewer requests, hook ingestion, and ordinary Claude Code traffic do not receive it.

CC Switch treats the header as display attribution, not authentication or authorization. It recognizes only the exact ASCII value `claude-mem`; any missing, empty, repeated, oversized, or different value receives the normal `proxy` source. This intentionally avoids a general-purpose client-controlled logging field.

For the recognized value, CC Switch:

1. captures `MEM` in request context before success or failure paths diverge;
2. removes the attribution header before constructing or forwarding the upstream request;
3. persists `MEM` through the normal usage logger's `data_source` field;
4. leaves `model`, `request_model`, `pricing_model`, token counts, costs, provider ID, session ID, and request body unchanged.

Success, upstream error, timeout, authentication failure, and other logged terminal outcomes retain the same source. Requests without the marker continue to rely on the database default `proxy` behavior.

The marker is deliberately not secret. A local process can imitate it, so the UI label is operational attribution rather than a security identity. The security requirement is that the marker never leaves the loopback CC Switch boundary.

## Repository boundaries

`satan9394/claude-mem_local` owns:

- emitting the exact header only for loopback CC Switch requests;
- tests proving model and request payload invariance;
- dual-upstream release detection and the compatibility registry;
- cross-repository contract tests or probes that can run without real provider credentials;
- documentation of the MEM label and upgrade policy.

The user-controlled CC Switch fork owns:

- parsing and consuming the exact attribution header;
- persisting `MEM` in proxy usage logs, including failure rows;
- preserving `proxy` for ordinary requests;
- tests proving the header is not forwarded and model/accounting fields are unchanged.

The official `farion1231/cc-switch` remote remains the read-only upstream. Local changes are not pushed to that official repository. If the CC Switch patch is published, it uses a separately named user-controlled fork and a `vX.Y.Z-local.N` version line so it cannot be mistaken for the official build.

## Dual-upstream registry

The single Claude-Mem upstream record is replaced by `.github/upstream-bases.json` with schema version `1`. It contains two explicit components:

```json
{
  "schemaVersion": 1,
  "components": {
    "claude-mem": {
      "repository": "thedotmack/claude-mem",
      "releaseTag": "v13.11.0",
      "commit": "fad1872bd985bc51e314286db4d90b018f5598c3",
      "reviewKind": "source-base"
    },
    "cc-switch": {
      "repository": "farion1231/cc-switch",
      "releaseTag": "v3.17.0",
      "commit": "3d176b98cc0bfd151a42882e88ab59b62083b92f",
      "reviewKind": "compatibility-base"
    }
  }
}
```

The registry records the last release that completed local review, not whichever tag was merely discovered. A detected update never mutates this file automatically.

## Release monitoring and review issues

The existing scheduled workflow becomes a generic, least-privilege release watcher. On its weekly schedule and manual dispatch, it reads the registry, resolves each component's latest stable GitHub Release, and compares tag and commit with the reviewed base.

For each changed component it creates or refreshes one deduplicated issue. The issue title includes the component and target tag. Its body includes:

- previous reviewed tag and commit;
- latest stable tag, commit, release URL, and publication time;
- a comparison URL;
- the component-specific security and compatibility checklist;
- a reminder that detection does not authorize merging, installation, tagging, or publishing.

Prereleases and drafts are ignored. A failure to resolve one component is reported without treating the other as current. The workflow retains only `contents: read` and `issues: write`; it receives no release, package, pull-request write, secret, or deployment permission.

Claude-Mem reviews use the existing local-security classification. CC Switch reviews focus on the proxy handler, request context, response processor, usage logger, database migrations, model mapper, pricing, and usage UI. Any change in those areas requires the MEM contract suite before the reviewed base advances.

## Compatibility gates

An upgrade is accepted only when automated evidence proves all of the following:

- ordinary Claude Code traffic records Source `proxy`;
- Claude-Mem traffic records Source `MEM`;
- both paths preserve the original `model` and `request_model` values exactly;
- provider selection, model mapping, pricing model, token counts, and cost calculation are unaffected by attribution;
- the attribution header is absent from the upstream request;
- success and logged failure outcomes retain correct attribution;
- malformed or unrecognized attribution values fall back to `proxy`;
- CC Switch health and `/v1/messages` compatibility still pass on loopback;
- Claude-Mem sends no real CC Switch provider key and does not access CC Switch's provider database;
- local-only startup and idle execution still produce no unapproved external traffic.

The suite uses a local capture server or in-process request harness and synthetic usage data. It does not require paid model calls or production credentials.

## Runtime version visibility

GitHub Releases are the authoritative source for upstream availability. Runtime compatibility is established by the protocol probe, not inferred solely from a binary filename or Windows file metadata.

The maintenance diagnostics report three independent states:

- reviewed upstream tag from the registry;
- latest detected upstream tag from GitHub Actions or a manual check;
- live protocol result for the locally running CC Switch instance.

Windows executable version metadata may be displayed as supplemental evidence when available, but missing or inaccurate metadata never bypasses the protocol gate. No broad version endpoint is added unless a future compatibility need cannot be met by the existing health and synthetic request probes.

## Failure handling

Attribution failure must not break model execution. If Claude-Mem cannot add the local marker, it reports a redacted diagnostic and the request follows the existing provider failure policy; it never changes the model to compensate. If CC Switch does not support the contract, the compatibility probe reports `MEM_LABEL_UNSUPPORTED`, and the UI may show `proxy`; model traffic remains otherwise unchanged.

Malformed source markers are ignored rather than echoed. Database write failures follow CC Switch's existing logging behavior and do not trigger a second direct database writer. GitHub API or workflow failures do not change either reviewed base and do not start synchronization work.

Unknown upstream changes, ambiguous model behavior, missing contract evidence, or any sign that the marker reaches an external provider fail the maintenance gate closed.

## Test strategy

Implementation follows test-driven development in each repository:

1. add a failing test for exact header emission and unchanged Claude-Mem request payload;
2. implement the minimum provider change and run focused and provider regression suites;
3. add failing CC Switch tests for recognized, absent, malformed, success, failure, and non-forwarded marker behavior;
4. implement the minimum request-context and logger changes and run Rust/backend regression tests;
5. add failing workflow tests for both registry components, deduplication, prerelease exclusion, partial API failure, and least-privilege permissions;
6. generalize the watcher and run infrastructure tests;
7. run a local end-to-end probe with one ordinary synthetic request and one Claude-Mem synthetic request, then verify Source, model, token, cost, and captured upstream headers.

No test writes to the user's live CC Switch database. Integration tests use temporary databases and loopback fixtures.

## Delivery and publication boundaries

Local implementation, tests, commits, and build artifacts may be prepared without changing installed applications or remote repositories. Replacing the installed CC Switch executable, creating a GitHub fork, pushing branches, opening pull requests, tagging, releasing, or publishing packages are separate external actions and require the user's final confirmation under the repository operating rules.

The implementation remains source-first. It does not introduce automatic self-update, automatic upstream merge, direct writes to another application's database, a second proxy, or a new package distribution channel.

## Acceptance criteria

The design is complete when:

- a normal Claude Code request and a Claude-Mem request can use the same original model without any model-name mutation;
- CC Switch visibly distinguishes only the Claude-Mem request as Source `MEM`;
- captured outbound traffic proves the local attribution header never reaches the provider;
- both upstream projects are monitored against reviewed release bases with deduplicated, actionable issues;
- upstream detection cannot merge, install, tag, release, or publish anything;
- focused, full, and end-to-end tests provide repeatable evidence for future upgrades;
- documentation makes the local modifications and compatibility boundary visible from the Claude-Mem Local repository.
